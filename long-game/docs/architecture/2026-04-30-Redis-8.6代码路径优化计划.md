## 目标

在**不引入集群特性、不改变业务行为、不牺牲现有兼容性**的前提下，围绕当前仓库最明确的 5 个 Redis 访问热点做定点优化：

1. `totalClickCount` 全量遍历排行榜
2. 背包实例读取 N+1
3. 热点 `HGetAll` 改 `HMGet`
4. 排名查询合并 `pipeline`
5. 仅评估天赋状态那条 `SCAN`

本计划只包含**代码改动方案**，不包含压测平台、监控面板、运维迁移或集群设计。

## 总原则

- 每一项优化都要保持对外接口、返回字段、排序规则、兼容逻辑不变。
- 每一项优化都应当能**单独提交、单独回归、单独回滚**。
- 优先新增或修改测试，再改实现。
- 如果某项优化无法在当前测试体系下证明“无行为变化”，则不进入实施。
- 所有 Redis 读写优化默认以“减少命令数 / 减少返回字段 / 减少 RTT / 降低 O(n) 扫描”为目标，不做顺手重构。

## 范围与非范围

### 范围内

- `backend/internal/vote/store.go`
- `backend/internal/vote/talent.go`
- `backend/internal/vote/store_test.go`
- 必要时补充小型测试文件，但尽量复用现有 `store_test.go`

### 明确不做

- 不改前端
- 不改 Redis 数据结构大类（例如把 Hash 改成 String、把现有 JSON 字段强行拆表）
- 不改 SSE / PubSub 架构
- 不改 Lua 主链路业务语义
- 不因为 Redis 8.6 升级而做风格型重构

## 实施顺序

按风险和收益排序，建议实施顺序如下：

1. `totalClickCount` 去掉全量遍历
2. 排名查询合并 `pipeline`
3. 热点 `HGetAll` 改 `HMGet`
4. 背包实例读取去 N+1
5. 天赋状态 `SCAN` 评估后再决定是否实施

前四项都是**低到中风险、收益明确**。第五项先做证据评估，不承诺本轮一定改。

## 任务 1：消除 `totalClickCount` 的全量排行榜遍历

状态：已完成（提交 `156438a`）

### 问题

当前 [backend/internal/vote/store.go](/Users/Learning/web/long/backend/internal/vote/store.go:1164) 的 `totalClickCount` 通过 `ZRangeWithScores` 拉取整个排行榜并在 Go 侧求和，复杂度为 O(n)。

只要排行榜人数持续增长，这条路径就会随着玩家规模线性变慢。

### 目标

把总点击数改成**单独维护的累加值**，读取时 O(1) 返回，同时保持现有对外语义不变。

### 拟改文件

- 修改：`backend/internal/vote/store.go`
- 修改：`backend/internal/vote/lua.go`
- 修改：`backend/internal/vote/store_test.go`

### 方案

新增一个总点击数 key（建议沿用现有 namespace，命名与仓库风格一致），在所有会增加点击数的路径里同步维护：

- 普通按钮点击
- Boss 点击 Lua 脚本
- 可能影响总点击数的挂机/自动点击路径

读取总点击数时优先读新 key；如 key 不存在，再走一次旧逻辑兜底，避免直接破坏旧数据兼容。

### 约束

- 不能改变总点击数定义
- 不能丢失 Lua 原子性
- 不能让 Boss 点击和普通点击口径不一致

### 验证

- 补测试覆盖普通点击后总数变化
- 补测试覆盖 Boss 点击后总数变化
- 补测试覆盖旧 key 不存在时的兼容读取
- 运行：`go -C backend test ./internal/vote/...`

### 目标

### 约束

- 不改变 rank 从 0 基转 1 基的口径

### 验证

- `currentBossFromCmdable`
- `getEquipmentDefinition`
- `recentRewardsForNickname`

#### 第二批：依赖关系稍复杂

- `getOwnedInstance`
- `itemInstancesByIDForNickname`
- `GetTalentState`

### 约束

- 不能丢字段
- 不能改变空 Hash / 缺字段 / 旧数据兼容行为
- Boss `parts` 仍需保留 JSON 反序列化逻辑

### 验证

## 任务 4：消除背包实例读取 N+1

状态：已完成（提交 `4518c9b`）

### 问题

当前 [backend/internal/vote/store.go](/Users/Learning/web/long/backend/internal/vote/store.go:2555) 的 `itemInstancesByIDForNickname` 是：

1. `SMembers` 拿全部实例 ID
2. 对每个实例逐个 `HGetAll`

这是典型 N+1 访问模式，实例数增长时 RTT 会线性增长。

### 目标

在不改变返回结构、排序、缺失实例容错行为的前提下，把实例详情读取改成**批量 pipeline**。

### 拟改文件

- 修改：`backend/internal/vote/store.go`
- 修改：`backend/internal/vote/store_test.go`

### 方案

保留第一步 `SMembers`；
第二步改为：

- 对每个 `equipmentInstanceKey(instanceID)` 发起同一批 pipeline 请求
- 单实例读取改为 `HMGet` 固定字段
- 汇总结果时继续保持：
  - 空实例跳过
  - `item_id` 为空跳过
  - `wrong type` 兼容语义不变

### 注意

`loadoutForNickname` 和 `inventoryForNickname` 都依赖这个函数，所以它的行为一旦变化会影响多条读取路径。该任务必须单独提交并重点测试。

### 验证

- 补测试覆盖多个实例批量读取
- 补测试覆盖其中某个实例 Hash 缺失时仍能返回剩余实例
- 补测试覆盖 `loadoutForNickname` / `inventoryForNickname` 的结果不变
- 运行：`go -C backend test ./internal/vote/...`

## 任务 5：仅评估天赋状态那条 `SCAN`

状态：未实施；已完成代码级评估，当前暂缓

### 问题

当前 [backend/internal/vote/store.go](/Users/Learning/web/long/backend/internal/vote/store.go:1808) 的流血 tick 处理会调用 [backend/internal/vote/store.go](/Users/Learning/web/long/backend/internal/vote/store.go:1857) `listTalentCombatStateNicknames`，其核心是：

- `SCAN vote:player:talent_state:*:<bossID>`

这条扫描不是 fallback，而是现行实现的一部分，所以与 `players:index` 那种兜底扫描不同。

### 本轮目标

本轮只做**代码级评估**，不默认实施替换。

### 当前评估结论

- 这条 `SCAN` 不是兜底逻辑，而是流血结算主路径的一部分
- 真正难点不在扫描本身，而在于若改为显式索引，需要把“按 key 自动发现参与者”改成“由应用维护参与者集合”
- 现有战斗状态以整块 JSON 存储，且存在多处写入/清理入口，索引一致性风险高于前四项
- 因此本轮先停在评估，不进入实现

### 评估输出要求

在实施前，需要先回答下面 3 个问题：

1. 当前这条 `SCAN` 的触发频率是否足以成为真实热点
2. 如果新增显式索引，写路径是否会显著变复杂
3. 现有测试体系是否足以证明“索引版实现无行为偏差”

### 可能的实施方向

如果评估后确认值得改，再进入下一轮计划：

- 为 `bossID -> active talent nicknames` 新增显式索引
- 在创建 / 更新 / 删除战斗状态时同步维护索引
- 将 `listTalentCombatStateNicknames` 从 `SCAN` 改为索引读取

### 本轮不做

- 本轮不直接新增索引
- 本轮不改流血循环逻辑
- 本轮不改 Redis key 命名结构
