# 冷数据迁移与 Mongo 主存方案

## 目标

将以下冷数据统一切换为 MongoDB 主存储：

- Boss 历史
- 留言墙历史
- 后台操作审计日志
- 系统日志与业务事件日志

切换完成后：

- 不再保留 Redis / Mongo 读源选择开关
- 以上冷数据固定写入 MongoDB
- 以上冷数据固定从 MongoDB 读取
- Redis 中对应历史冷数据在迁移校验通过后直接删除

## 迁移范围

## 需要迁移的历史数据

这次只有两类存在 Redis 存量，需要做历史迁移：

- Boss 历史
- 留言墙历史

## 不需要迁移的历史数据

以下两类当前没有 Redis 现成历史 key，不做历史补迁，只从切换上线时开始写 Mongo：

- 后台操作审计日志
- 系统日志与业务事件日志

## 最终数据归属

### MongoDB

- `boss_history`
- `wall_messages`
- `admin_audit_logs`
- `domain_events`
- `system_logs`

### Redis

继续保留：

- 按钮计数与按钮定义热路径
- Boss 当前战斗态
- 排行榜
- 玩家登录态与实时状态
- SSE 实时依赖数据

迁移完成后删除：

- Boss 历史冷数据 key
- 留言墙历史冷数据 key

## 方案选择

采用独立 Go 迁移工具方案，而不是一次性脚本。

原因：

- 支持正式服重复执行
- 支持 `plan / migrate / verify / cleanup` 分阶段执行
- 迁移失败后容易定位
- 先验后删，降低误删 Redis 的风险

## 工具形态

建议新增：

- `backend/cmd/migratecolddata`

命令子模式：

- `plan`
- `migrate`
- `verify`
- `cleanup`

配置来源保持与服务一致：

- 通过 `CONSUL_ADDR`
- 通过 `CONSUL_CONFIG_KEY`

也就是说，迁移工具与服务端读取同一份 Consul YAML，其中必须包含：

- `redis`
- `redis_prefix`
- `mongo`

## 执行流程

正式服和测试服统一按下面顺序执行：

1. 停服务
2. 执行 `plan`
3. 执行 `migrate`
4. 执行 `verify`
5. 人工确认结果
6. 执行 `cleanup`
7. 启动新版本服务

## plan 阶段

目标：

- 统计 Redis 中待迁移的 Boss 历史数量
- 统计 Redis 中待迁移的留言数量
- 输出预计写入 Mongo 的文档数
- 输出会删除的 Redis key 数量与 key 模式

要求：

- 只读，不写 Mongo，不删 Redis
- 输出摘要和明细统计

## migrate 阶段

目标：

- 将 Boss 历史写入 `boss_history`
- 将留言墙历史写入 `wall_messages`

要求：

- 必须幂等
- 允许重复执行
- 默认使用 upsert，而不是盲目 insert
- 单条失败要记录并汇总，不能静默跳过

## verify 阶段

目标：

- 核对 Redis 历史数量和 Mongo 文档数量
- 对 Boss 历史做关键字段抽样核对
- 对留言墙历史做关键字段抽样核对

至少校验：

- 数量一致
- 主键一致
- 关键时间字段一致
- 关键内容字段一致

## cleanup 阶段

目标：

- 仅删除已经迁移并校验通过的 Redis 冷数据

要求：

- 删除逻辑必须按明确 key 模式执行
- 不允许模糊删除
- 删除前打印即将删除的 key 范围
- 删除后打印删除统计

## Redis 删除边界

## Boss 历史

删除：

- `vote:boss:history`
- `vote:boss:history:*`

实际 key 名需基于 `redis_prefix` 推导，不写死 `vote:`

## 留言墙历史

删除：

- 留言有序集合 key
- 留言明细 key 前缀
- 仅限留言历史相关 key

实际删除范围要以现有 `Store` 里的留言 key 生成规则为准。

## 明确不能删除

- 当前 Boss
- 玩家资源
- 玩家背包
- 排行榜
- 按钮计数
- 公告
- JWT 与会话
- 任何与实时前台直接相关的 key

## Mongo 集合要求

## boss_history

建议字段：

- `boss_id`
- `template_id`
- `name`
- `status`
- `max_hp`
- `current_hp`
- `gold_on_kill`
- `stone_on_kill`
- `talent_points_on_kill`
- `parts`
- `started_at`
- `defeated_at`
- `loot`
- `damage`
- `archived_at`

## wall_messages

建议字段：

- `message_id`
- `nickname`
- `content`
- `created_at`
- `status`

## admin_audit_logs

切换后新增写入，字段至少包括：

- `operator`
- `action`
- `target_type`
- `target_id`
- `request_path`
- `request_ip`
- `result`
- `error_code`
- `created_at`

## domain_events

切换后新增写入，字段至少包括：

- `event_type`
- `nickname`
- `boss_id`
- `item_id`
- `payload`
- `created_at`

## system_logs

切换后新增写入，字段至少包括：

- `level`
- `module`
- `message`
- `fields`
- `request_id`
- `created_at`

## 应用层改造要求

## Boss 历史

- 新写路径固定写 Mongo
- 后台 Boss 历史固定读 Mongo
- 公共 Boss 历史接口固定读 Mongo
- Redis 历史写入逻辑删除

## 留言墙历史

- 创建留言改为写 Mongo
- 留言列表改为读 Mongo
- 删除留言改为删 Mongo
- Redis 留言历史读写逻辑删除

## 审计日志

- 后台关键写操作接入 `admin_audit_logs`
- 审计写失败不影响主业务成功响应，但必须记错误日志

## 系统/业务事件日志

- Boss 激活、Boss 击杀、掉落结算、密码重置、装备关键操作接入事件日志
- 高价值错误写入 `system_logs`

## 风险控制

## 停机窗口

因为测试服允许短暂停机，且正式服脚本也按停机迁移设计，所以这次不做在线增量迁移。

## 幂等要求

迁移工具必须支持重复执行，避免以下问题：

- 中途失败后无法重跑
- 已写入部分数据后再次运行造成重复

## 删除保护

`cleanup` 不能和 `migrate` 自动串联。必须人工确认 `verify` 通过后再执行。

## 回滚边界

迁移完成且 Redis 冷数据已删后，不提供 Redis 级快速回退。

因此回滚只限于：

- 应用版本回滚
- Mongo 数据保留

也就是说，正式执行 `cleanup` 前必须完成校验。

## 推荐实施顺序

1. 先补齐 Mongo 下的留言仓储
2. 再实现迁移工具 `plan / migrate / verify / cleanup`
3. 再改应用读写路径到 Mongo 主存
4. 再接入后台审计日志
5. 再接入系统/业务事件日志
6. 最后做测试服停机迁移演练

## 结论

这次迁移项目的核心原则是：

- Redis 留实时
- Mongo 管冷数据
- 历史迁移只做 Boss 历史和留言墙历史
- 审计和事件日志从切换时刻开始写 Mongo
- 停机迁移、先验后删、删除后不回退 Redis
