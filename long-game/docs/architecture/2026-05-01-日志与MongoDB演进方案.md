# 日志与 MongoDB 演进方案

## 背景

当前后端运行时日志主要依赖 Go 标准库 `log`，日志点数量较少，且以字符串拼接输出为主。根据现有代码，日志主要集中在：

- 服务启动与退出：`backend/cmd/server/main.go`
- Consul 配置监听：`backend/internal/config/config.go`
- AFK 结算与恢复链路：`backend/internal/httpapi/afk_service.go`

同时，项目当前所有业务数据都放在 Redis 中。对实时状态来说这是合适的，但对于历史 Boss、留言归档、审计记录、运营日志这类冷数据，继续长期堆在 Redis 中会带来几个问题：

- Redis 内存成本高，不适合长期保留大量历史记录
- 历史列表常常需要“按时间查、分页查、按条件筛”，这不是 Redis 的强项
- 现有部分分页实际上是“先把全量历史读出来，再在 Go 内存里分页”，随着数据增长会变差

这份文档给出一个务实的两段式方案：

1. 先补齐结构化日志能力，提升线上可观测性
2. 再把明确的冷数据与日志数据分层到 MongoDB

## 结论摘要

### 1. 是否考虑使用 zap

建议使用，但应当控制范围，先只替换后端运行时日志，不做全仓库一把梭重构。

原因：

- 当前项目是 Go 服务，`zap` 已经是成熟、低开销、生态稳定的结构化日志方案
- 现有标准库 `log` 输出不利于后续接入日志采集、检索、告警与字段过滤
- 后续如果 MongoDB 要承接部分业务日志或审计日志，日志字段结构必须先规范，否则落库价值不高

不建议的做法：

- 不建议为了“日志统一”把所有业务事件都直接同步写 Mongo
- 不建议一开始就把 `zap` 深度抽象成复杂日志框架
- 不建议第一阶段就把每个函数都加日志，噪音会大于收益

### 2. 是否使用 MongoDB 承接冷数据和日志

建议使用，但只承接“历史型、查询型、审计型”数据，不承接当前高频实时主状态。

推荐定位：

- Redis：实时主存储、计数、排行榜、会话、SSE 热状态
- MongoDB：冷数据归档、历史查询、审计记录、结构化业务日志

不建议迁入 MongoDB 的数据：

- 按钮实时计数
- Boss 当前血量与当前战斗态
- 在线玩家实时状态
- 高频点击主链路中的同步写数据

这些仍应留在 Redis，否则会把当前“快路径”变成“双写热点路径”，风险太高。

## 现状判断

## 日志现状

当前日志能力有 4 个明显缺口：

1. 缺少统一 logger 注入点  
   目前多处直接使用 `log.Printf`，很难按模块统一控制字段、级别与输出格式。

2. 缺少日志级别  
   现在基本只有“打印”与“不打印”，缺少 `debug / info / warn / error` 分层。

3. 缺少请求与业务上下文  
   例如 `nickname`、`bossId`、`trace/request id`、接口路径、耗时、错误码等字段没有统一附带。

4. 缺少可扩展输出面  
   现在更偏向控制台文本输出，不利于后续进入 ELK、Loki、Mongo 或其他检索系统。

## 数据现状

从当前代码看，已经有一些明显的冷数据候选：

- Boss 历史：`backend/internal/vote/admin_history.go`
- Boss 历史分页：`backend/internal/vote/admin_pages.go`
- 留言墙历史消息
- 公告/运营内容历史版本
- 后台管理操作审计
- 配置变更、任务执行、异常恢复等系统日志

其中 Boss 历史最有代表性：当前仍保存在 Redis，且后台分页会先读取全量历史再在 Go 里分页。这种模式在数据量小时可接受，但不适合作为长期形态。

## 总体设计

## 设计原则

- 不改变 Redis 作为实时主存储的定位
- 不把 MongoDB 引入到高频点击同步主链路
- 所有新能力都应支持“关闭开关后退回当前行为”
- 优先解决可观测性，再做存储分层
- 先做双写和只读旁路，再决定是否切主查询

## 目标架构

### 日志层

后端统一引入一个轻量日志包，例如：

- `backend/internal/xlog/`

职责：

- 初始化 `zap.Logger`
- 根据配置选择开发格式或 JSON 格式
- 提供基础字段注入能力
- 提供从 `context.Context` 补充请求字段的方法

推荐输出策略：

- 开发环境：控制台友好格式
- 生产环境：JSON 结构化日志

### 存储分层

建议形成三层职责：

1. Redis 热数据层  
   放实时计数、实时状态、排行榜、会话、SSE 快照依赖数据

2. MongoDB 冷数据层  
   放历史归档、审计记录、可筛选的业务事件、可回溯日志

3. 进程日志输出层  
   `zap` 先输出到 stdout/stderr，后续再由外部采集；如确有需要，再增加“部分日志异步入 Mongo”的能力

## MongoDB 数据分类建议

### 第一批适合进入 MongoDB

#### 1. Boss 历史归档

建议优先迁这类数据，因为：

- 天然按时间增长
- 查询明显偏历史回看
- 有后台分页和筛选需求
- 当前 Redis 方案已经开始显露“全量读取后分页”的问题

建议集合：

- `boss_history`

建议字段：

- `boss_id`
- `template_id`
- `name`
- `status`
- `started_at`
- `defeated_at`
- `max_hp`
- `current_hp`
- `gold_on_kill`
- `stone_on_kill`
- `talent_points_on_kill`
- `parts`
- `loot`
- `damage_top`
- `archived_at`

索引建议：

- `{ started_at: -1 }`
- `{ boss_id: 1 }` 唯一
- `{ template_id: 1, started_at: -1 }`

#### 2. 留言墙历史

留言具备明显的内容检索与审查需求，放 Mongo 比 Redis 更合适。

建议集合：

- `wall_messages`

建议字段：

- `message_id`
- `nickname`
- `content`
- `created_at`
- `status`
- `review_flags`

索引建议：

- `{ created_at: -1 }`
- `{ nickname: 1, created_at: -1 }`
- 如要做后台关键词检索，可额外评估文本索引

#### 3. 后台操作审计

这是最适合 Mongo 的一类数据，且对排障价值很高。

建议集合：

- `admin_audit_logs`

建议字段：

- `operator`
- `action`
- `target_type`
- `target_id`
- `request_path`
- `request_ip`
- `payload_summary`
- `result`
- `error_code`
- `created_at`

索引建议：

- `{ created_at: -1 }`
- `{ operator: 1, created_at: -1 }`
- `{ action: 1, created_at: -1 }`

### 第二批可考虑进入 MongoDB

#### 4. 业务事件日志

例如：

- Boss 激活/击杀/掉落结算
- 装备生成、重铸、分解
- 玩家密码重置
- 循环 Boss 启停

这类日志更适合作为“事件流归档”而不是主业务表。

建议集合：

- `domain_events`

建议字段：

- `event_type`
- `nickname`
- `boss_id`
- `item_id`
- `payload`
- `created_at`

#### 5. 系统异常与任务执行日志

如果后续希望在后台中直接查询错误记录，可以把部分高价值错误异步落 Mongo。

建议集合：

- `system_logs`

建议字段：

- `level`
- `module`
- `message`
- `fields`
- `request_id`
- `created_at`

注意：这类日志不应全量、无节制落库，否则会把 Mongo 变成低配日志平台。建议只接收：

- `warn` / `error`
- 关键运营动作日志
- 需要后台检索的审计日志

## 明确不建议首批迁移的数据

- 按钮定义和按钮计数主数据
- 玩家实时战斗状态
- Boss 当前状态
- 实时排行榜
- JWT 会话与登录态
- 高频挂机 Tick 中的细粒度过程状态

原因很直接：这些数据要么高频写、要么强依赖 Redis 现有结构、要么直接服务 SSE 快照，不适合现在切走。

## zap 引入方案

## 目标

在不改业务行为的前提下，把运行日志从“零散文本打印”升级成“可分级、可检索、可扩展的结构化日志”。

## 推荐落地方式

### 阶段 1：只建立统一 logger，不追求全覆盖

拟调整范围：

- `backend/cmd/server/main.go`
- `backend/internal/config/config.go`
- `backend/internal/httpapi/afk_service.go`
- 新增 `backend/internal/xlog/`

建议做法：

- 新增日志配置结构，例如 `log.level`、`log.format`、`log.sample`、`log.include_caller`
- 在启动入口初始化全局基础 logger
- 先把已有 `log.Printf` 改成 `zap` 的结构化输出
- 先覆盖启动、关闭、配置监听失败、AFK 异常这几类高价值日志

预期收益：

- 能按字段检索错误
- 能区分 `info`、`warn`、`error`
- 为后续请求日志、中间件日志、Mongo 日志归档打基础

### 阶段 2：补请求维度日志

拟调整范围：

- `backend/internal/httpapi/`

建议做法：

- 增加 Hertz 中间件，记录请求方法、路径、状态码、耗时、客户端 IP
- 为管理接口和玩家写接口补充 `request_id`
- 对关键写接口补“成功/失败的摘要日志”，不是全量 body 日志

注意：

- 不记录密码、JWT、完整敏感请求体
- 不在每次点击接口打印一条 `info`，否则日志量会失控

### 阶段 3：补业务审计日志

这一步不要和普通运行日志混在一起。

建议把“审计日志”视为单独能力，例如：

- 管理员登录
- 修改 Boss 模板
- 开关 Boss 循环
- 重置玩家密码
- 人工发奖励或改配置

这些记录更适合写入 MongoDB 的 `admin_audit_logs`，而不是只打 stdout。

## MongoDB 引入方案

## 目标

让 Redis 回归实时态，让 MongoDB 承担历史查询和审计沉淀，但不破坏当前主链路稳定性。

## 推荐落地路径

### 阶段 1：先接入基础设施，不切读流量

拟新增：

- `backend/internal/mongostore/`
- 配置中的 `mongo` 段

建议配置字段：

```yaml
mongo:
  enabled: false
  uri: "mongodb://127.0.0.1:27017"
  database: "vote_wall"
  connect_timeout_ms: 3000
  write_timeout_ms: 2000
```

第一阶段只做：

- 建连
- 健康检查
- 仓储接口定义
- 关闭开关时完全不影响当前逻辑

### 阶段 2：Boss 历史双写

以 Boss 历史作为首个试点。

建议做法：

- 保持 Redis 现有写入不动
- 在 `SaveBossToHistory` 成功后，异步或旁路写 Mongo
- 写失败只记日志和指标，不影响主流程

### 异步写 Mongo 的三种实现方案

目标不是“所有写 Mongo 都用 `go func()`”，而是“Mongo 写入不阻塞 Redis 主链路”。围绕这个目标，建议把实现方案分成三档。

#### 方案 A：裸 `go func()` 异步写

做法：

- Redis 主写成功后，直接起一个 `go func()` 执行单次 Mongo 写入
- 写入时自带超时 `context`
- 失败后只记日志，不回传给主请求

优点：

- 实现最简单
- 适合快速验证链路
- 对非常低频的操作接入成本低

缺点：

- 并发不可控，流量上来后容易 goroutine 膨胀
- 不方便做排队、限流、重试和优雅停机
- 一旦 Mongo 抖动，失败会集中放大

适用场景：

- 初期 PoC
- 极低频后台操作审计
- 本地开发或短期试验

结论：

- 可以作为最早期验证方案
- 不建议作为 Boss 历史双写或长期日志归档的正式方案

#### 方案 B：进程内 channel 队列 + 固定 worker

做法：

- 主链路只负责把归档任务投递到进程内 `channel` 缓冲队列
- 后台启动固定数量 worker goroutine，从 `channel` 消费并执行 Mongo 单条写入
- 每条写入都有独立超时
- 队列满时走明确策略：丢弃、降级为只记日志、或统计失败次数
- 这里的“缓冲队列”特指进程内内存队列，不是 Redis Stream

优点：

- 并发上限可控，不会无限起 goroutine
- 容易加监控，例如队列长度、丢弃数、失败数、处理耗时
- 容易做优雅停机和收尾
- 实现复杂度适中，足够覆盖大多数冷数据双写需求

缺点：

- 比裸 `go func()` 多一层队列与 worker 管理
- 仍然是单条写入，Mongo 写放大还在

适用场景：

- Boss 历史双写
- 后台操作审计
- 中低频业务事件归档

结论：

- 这是当前项目最推荐的默认方案
- 能较好平衡主链路安全、实现复杂度与可运维性

#### 方案 C：进程内 channel 队列 + 批量聚合 flush

做法：

- 主链路把事件投递到进程内 `channel` 缓冲队列
- 后台聚合器 goroutine 从 `channel` 取任务，按“条数阈值”或“时间窗口”攒批
- 使用 Mongo `InsertMany` 或批量写接口统一落库
- 失败时按整批或拆批策略处理

优点：

- Mongo 写入效率更高，适合中高量历史事件
- 可显著降低每条事件的单次写放大
- 更适合未来日志量持续增长的场景

缺点：

- 实现最复杂
- 会引入更高的短时延迟
- 批次失败后的重试、去重、拆分处理更麻烦
- 优雅停机时需要 flush 未落库缓存

适用场景：

- 中高频业务事件归档
- 量比较大的系统日志沉淀
- 未来确认 Mongo 承接的历史量明显增长后

结论：

- 不建议作为第一版就上
- 适合作为方案 B 跑稳之后的后续优化方向

### 推荐选择

建议按下面顺序使用：

1. 验证期：方案 A  
   只用来快速证明 Mongo 模型与链路可行。

2. 首个正式版本：方案 B  
   作为 Boss 历史双写、后台审计日志的默认实现。

3. 数据量明显增长后：方案 C  
   当单条写入成本和日志量都开始变成问题时，再演进到批量 flush。

### 当前项目的建议落点

结合当前项目规模与风险控制要求，建议：

- Boss 历史双写：优先方案 B
- 后台操作审计：低频时可先 A，正式化后切 B
- 系统错误日志入 Mongo：如未来量变大，再考虑 C

换句话说，本项目不建议把“异步”直接等同于“每次都 `go func()`”；更建议把异步写 Mongo 做成一个可控的后台投递能力。当前文档里提到的“缓冲队列”默认都指进程内 `channel` 队列，不指 Redis Stream。

推荐原因：

- 这是冷数据
- 不要求强事务
- 允许短时间 Redis 与 Mongo 之间存在轻微延迟

这一阶段验证的是：

- Mongo 模型是否合适
- 运维连接是否稳定
- 写入失败是否可观测

### 阶段 3：后台 Boss 历史查询切到 Mongo

在双写稳定后，再把：

- `/api/admin/boss/history`

改为优先查 Mongo。

建议保留回退机制：

- Mongo 不可用时，回退 Redis 查询
- 或通过配置开关决定读源

这样可以把风险控制在“查询路径”而不是“写入主链路”。

### 阶段 4：扩展到留言归档与审计日志

Boss 历史跑稳后，再做第二批：

- 留言墙历史
- 后台操作审计

顺序上优先审计日志，其次留言归档。原因是审计日志结构更稳定、回报更直接。

## 模块拆分建议

建议在后端保留明确边界，避免 Redis 与 Mongo 逻辑交织到业务代码里。

推荐职责划分：

- `internal/vote/`：保持核心业务与 Redis 热数据主逻辑
- `internal/mongostore/`：Mongo 连接、集合访问、索引初始化、冷数据读写
- `internal/xlog/`：日志初始化与封装
- `internal/httpapi/`：只依赖抽象接口，不关心底层是 Redis 还是 Mongo

如果后续需要更清晰的双写编排，可增加很薄的一层归档服务，例如：

- `internal/archive/`

但首版不建议过度抽象，先保持最小改动。

## 配置与运维建议

## 配置项建议

在现有 Consul YAML 中新增：

```yaml
log:
  level: "info"
  format: "json"
  include_caller: false

mongo:
  enabled: false
  uri: "mongodb://127.0.0.1:27017"
  database: "vote_wall"
  connect_timeout_ms: 3000
  write_timeout_ms: 2000
  read_timeout_ms: 3000
```

同时建议增加业务开关：

```yaml
archive:
  boss_history_dual_write: false
  boss_history_read_source: "redis" # redis | mongo
  admin_audit_enabled: false
```

这样每一步都能灰度，不需要一次性切换。

## 索引与生命周期

Mongo 引入后，需要一开始就明确索引和保留策略。

建议：

- Boss 历史长期保留，按时间索引
- 审计日志至少保留 90 天，视运营要求调整
- 系统错误日志如写 Mongo，可加 TTL，避免无限增长

## 风险与取舍

## 引入 zap 的风险

- 需要统一 logger 传递方式，首轮会碰到一部分函数签名设计问题
- 如果一开始打点过多，会制造日志噪音
- 若日志字段命名不统一，后续检索价值会打折

控制方式：

- 首轮只改已有关键日志点
- 先规定字段名，如 `nickname`、`boss_id`、`request_id`、`module`
- 不做大规模 debug 日志泛滥

## 引入 Mongo 的风险

- 增加一套运维依赖
- 如果误把高频热路径同步写 Mongo，会拖慢请求
- 双写期间会出现短暂不一致，需要设计容忍策略
- 如果 schema 设计随意，后续检索和维护会很乱

控制方式：

- 首批只做冷数据
- 首批只做双写，不先切主
- 写 Mongo 默认异步/旁路失败不阻塞主流程
- 每个集合先定字段和索引，再落代码

## 建议实施顺序

### 第 1 阶段：日志基础设施

- 引入 `zap`
- 配置中增加 `log` 段
- 替换现有关键 `log.Printf`
- 增加基础请求日志中间件

验收标准：

- 启动、退出、配置监听失败、AFK 异常均输出结构化日志
- 生产模式可输出 JSON
- 本地开发可读性不明显变差

### 第 2 阶段：Mongo 基础接入

- 增加 `mongo` 配置
- 增加 Mongo 客户端初始化
- 增加 `mongostore` 基础仓储层
- 增加连接失败与超时日志

验收标准：

- 关闭 `mongo.enabled` 时行为与当前完全一致
- 开启后可完成建连与健康检查

### 第 3 阶段：Boss 历史双写与后台读切换

- `SaveBossToHistory` 双写 Redis + Mongo
- 增加后台 Boss 历史从 Mongo 读取的开关
- 保留 Redis 回退

验收标准：

- Boss 历史可以稳定写入 Mongo
- 后台分页查询不再依赖先全量读 Redis 再内存分页

### 第 4 阶段：审计日志

- 后台关键写操作写入 `admin_audit_logs`
- 后续视需要提供后台查询接口

验收标准：

- 管理操作具备可追溯性
- 关键操作可按时间和操作者检索

### 第 5 阶段：留言与其他冷数据归档

- 迁留言历史
- 评估公告历史、运营事件历史

这一步放最后，避免同时改太多业务面。

## 建议先做与暂缓事项

## 建议立即启动

- 引入 `zap`
- 规划 `log` 配置结构
- 接入 Mongo 基础设施
- 选择 Boss 历史做首个冷数据试点

## 建议暂缓

- 不要把点击日志全量落 Mongo
- 不要把所有 Redis 业务表迁到 Mongo
- 不要在第一版做复杂日志平台化
- 不要在第一版引入分布式追踪全家桶

## 最终建议

如果目标是“当前项目日志太少，且未来有冷数据和日志归档需求”，那么最稳妥的路线是：

1. 先用 `zap` 把运行日志结构化
2. 再用 MongoDB 承接 Boss 历史和后台审计这两类最明确的冷数据
3. 等双写与查询跑稳后，再扩展到留言等其他历史数据

这条路线的关键点不是“把技术都加进来”，而是严格区分：

- Redis 负责实时
- MongoDB 负责历史
- `zap` 负责可观测性

只要不把 Mongo 写入塞进高频点击主链路，这个演进方向是合理的。
