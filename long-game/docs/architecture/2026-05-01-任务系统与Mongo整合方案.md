# 任务系统与 Mongo 整合方案

## 1. 目标

在现有 Redis Vote Wall 项目中新增一套可由后台配置的任务系统，支持：

- 日常任务
- 周常任务
- 限时任务

任务奖励支持四类：

- 装备
- 金币
- 强化石
- 天赋点

第一版任务条件支持：

- 当天点击数量要求
- 周点击数量要求
- 击败 Boss 数量要求
- 强化次数要求

设计目标：

- 热路径不拖慢点击 / Boss / 强化主链路
- 后台可配置、可上下线、可复制任务
- 玩家前台可查看任务、进度和领取状态
- 周期结束后能归档到 Mongo，能查出哪些玩家完成、未完成、未参与

## 2. 总体架构

任务系统采用 **Mongo 管配置与归档，Redis 管热进度** 的混合方案。

### 2.1 Mongo 负责

- 任务定义
- 任务领取日志
- 任务周期归档
- 后台任务操作审计
- 玩家任务关键事件日志

### 2.2 Redis 负责

- 当前生效任务索引
- 玩家当前周期任务进度
- 玩家当前周期行为计数器
- 当前可领取状态
- 当前周期参与者集合

### 2.3 原因

- 任务定义、领取日志、周期归档属于冷数据，适合放 Mongo
- 点击、Boss、强化属于热路径，不能每次操作都直接写 Mongo
- 当前项目已经完成 Mongo 冷数据基础设施和异步队列，可直接复用

## 3. 任务规则

### 3.1 投放范围

第一版任务为 **全体玩家统一生效**。

### 3.2 周期规则

- 日常任务：北京时间 `Asia/Shanghai` 自然日，00:00 刷新
- 周常任务：北京时间自然周，周一 00:00 刷新
- 限时任务：按后台配置 `start_at` 到 `end_at` 自动生效与失效

### 3.3 条件结构

第一版每条任务为 **单条件单目标**。

不支持：

- 多条件同时满足
- AND / OR 条件组

### 3.4 奖励规则

单条任务可配置多奖励并列：

- 金币
- 强化石
- 天赋点
- 装备奖励列表

装备奖励第一版仅支持：

- 选择已有装备模板 `item_id`
- 配置固定数量 `quantity`
- 发放到玩家背包时按当前装备实例生成规则创建新实例

不支持：

- 随机装备池
- 指定强化等级
- 指定特殊品质

### 3.5 完成与领取

- 任务完成后进入 `completed_unclaimed`
- 奖励由玩家 **手动领取**
- 领取后变为 `claimed`
- 同一周期同一任务只允许领取一次

### 3.6 过期规则

- 日常 / 周常 / 限时任务到期后，未领取奖励直接作废
- 到期后的周期结果必须先归档到 Mongo，再允许 Redis 热数据清理

### 3.7 编辑规则

- 草稿任务可自由编辑
- 已生效任务禁止修改条件、周期类型、时间窗口、目标值
- 若需改规则，后台通过“复制任务”生成新草稿，再上线新任务

## 4. 数据模型

## 4.1 Mongo 集合

### `task_definitions`

字段建议：

- `task_id`
- `title`
- `description`
- `task_type`：`daily | weekly | limited`
- `status`：`draft | active | inactive | expired`
- `condition_kind`：`daily_clicks | weekly_clicks | boss_kills | enhance_count`
- `target_value`
- `rewards`
- `display_order`
- `start_at`
- `end_at`
- `created_at`
- `updated_at`

索引建议：

- `{ task_id: 1 }` 唯一
- `{ status: 1, task_type: 1, display_order: 1 }`
- `{ start_at: 1, end_at: 1 }`

### `task_claim_logs`

字段建议：

- `task_id`
- `cycle_key`
- `nickname`
- `rewards`
- `claimed_at`

索引建议：

- `{ nickname: 1, task_id: 1, cycle_key: 1 }` 唯一
- `{ claimed_at: -1 }`

### `task_cycle_archives`

存周期聚合结果。

字段建议：

- `task_id`
- `cycle_key`
- `task_type`
- `condition_kind`
- `target_value`
- `start_at`
- `end_at`
- `participants_total`
- `completed_total`
- `claimed_total`
- `expired_unclaimed_total`
- `unfinished_total`
- `not_participated_total`
- `archived_at`

索引建议：

- `{ task_id: 1, cycle_key: 1 }` 唯一
- `{ archived_at: -1 }`

### `task_cycle_player_results`

存周期逐玩家结果。

字段建议：

- `task_id`
- `cycle_key`
- `nickname`
- `progress`
- `target_value`
- `status`：`claimed | completed_unclaimed | unfinished | not_participated`
- `completed_at`
- `claimed_at`
- `archived_at`

索引建议：

- `{ task_id: 1, cycle_key: 1, nickname: 1 }` 唯一
- `{ task_id: 1, cycle_key: 1, status: 1 }`

## 4.2 Redis 热键

### 生效任务索引

- `task:active:index`

保存当前生效任务 ID 集合。

### 玩家任务进度

- `task:progress:{nickname}:{taskID}:{cycleKey}`

字段建议：

- `progress`
- `target_value`
- `status`
- `completed_at`

### 玩家可领取任务集合

- `task:claimable:{nickname}`

保存当前可领取的任务实例标识，例如 `{taskID}:{cycleKey}`。

### 玩家周期计数器

- `task:counter:daily_clicks:{nickname}:{yyyy-mm-dd}`
- `task:counter:weekly_clicks:{nickname}:{yyyy-week}`
- `task:counter:boss_kills:{nickname}:{cycleKey}`
- `task:counter:enhance_count:{nickname}:{cycleKey}`

### 周期参与者集合

- `task:participants:{taskID}:{cycleKey}`

用于归档时识别参与过该周期任务统计的玩家。

## 5. Redis TTL 与归档规则

Redis 键必须设置 TTL，但 TTL 只是兜底清理，不是事实存储。

### 5.1 TTL 建议

- 日常计数器 / 进度 / participants：保留 `2-3 天`
- 周常计数器 / 进度 / participants：保留 `8-10 天`
- 限时任务进度：保留到 `end_at + 1-3 天`
- `task:claimable:{nickname}`：整体可设 `30 天` 兜底 TTL，集合项在领取或过期时移除

### 5.2 归档优先原则

任务周期过期时：

1. 先查 Mongo 是否已归档
2. 未归档则从 Redis 读取该周期热数据
3. 写入 Mongo 周期汇总和玩家明细
4. 归档成功后才允许 Redis 过期或清理

### 5.3 扫描延迟场景

若昨日任务已过期，今天扫描才命中：

- 只要 Redis TTL 缓冲仍在，就执行补归档
- 若归档已存在，则直接跳过
- 若 Redis 热数据已丢失，则只能基于 `task_claim_logs` 做降级恢复，无法准确还原未领取完成和未完成状态

因此要求：

- Redis TTL 缓冲必须大于归档扫描的最大延迟

## 6. 进度统计与事件接入

当前项目没有现成的自然日 / 自然周任务计数器，因此必须新增统一的任务事件推进层。

建议新增统一内部接口：

- `RecordTaskEvent(nickname, eventKind, delta, eventTime)`

第一版事件类型：

- `click`
- `boss_kill`
- `equipment_enhance`

### 6.1 点击任务

接入点：

- 普通按钮点击
- Boss 部位点击

行为：

- 更新对应日常 / 周常点击计数器
- 更新命中的任务进度
- 达标则标记为可领取

### 6.2 Boss 击败任务

接入点：

- Boss 击杀结算逻辑

行为：

- 为合格玩家或击杀归属玩家累计 `boss_kill` 事件
- 更新命中的击败 Boss 任务进度

具体采用“所有参与结算玩家”还是“最终击杀归属玩家”，实现前需统一按现有 Boss 奖励归属逻辑收口，第一版建议与 Boss 奖励发放对象保持一致。

### 6.3 强化任务

接入点：

- 装备强化成功逻辑

行为：

- 累计 `equipment_enhance`
- 更新命中的强化任务进度

## 7. 领取链路

建议新增统一内部接口：

- `ClaimTaskReward(nickname, taskID, cycleKey)`

处理顺序：

1. 校验任务实例是否存在且仍可领取
2. 校验是否已领取
3. 发放金币 / 强化石 / 天赋点
4. 发放装备实例
5. 写 Mongo `task_claim_logs`
6. 更新 Redis 进度状态和 `claimable` 集合
7. 写 `domain_events`

要求：

- 领取必须幂等
- 并发领取只能成功一次

## 8. 周期归档服务

建议新增统一内部接口：

- `ArchiveExpiredTaskCycles(now)`
- `ArchiveTaskCycle(taskID, cycleKey)`

### 8.1 主触发方式

后台定时扫描：

- 每分钟或每 5 分钟扫描一次“已过期但未归档”的任务周期

### 8.2 兜底方式

当玩家侧或后台读到过期周期时：

- 若发现该周期未归档，则同步补归档

### 8.3 归档过程

1. 读取 `participants`
2. 读取玩家进度
3. 查询 `task_claim_logs`
4. 以 `players:index` 为全体玩家全集
5. 生成：
   - 周期聚合汇总
   - 周期逐玩家结果
6. 归档写入 Mongo
7. 标记该周期已归档

“未参与玩家”口径：

- 在 `players:index` 中，但不在当前周期 `participants` 中
- 记为 `not_participated`

## 9. 后台功能

后台新增“任务管理”Tab。

第一版能力：

- 任务列表
- 创建任务
- 编辑草稿任务
- 上线任务
- 下线任务
- 复制任务
- 删除草稿任务
- 查看周期归档汇总
- 查看某个周期的玩家结果明细

建议支持筛选：

- 日常 / 周常 / 限时
- 草稿 / 生效中 / 已下线 / 已过期
- 周期结果状态：已领取 / 完成未领取 / 未完成 / 未参与

后台关键操作必须写入现有 `admin_audit_logs`。

## 10. 玩家前台功能

前台新增任务面板，展示：

- 当前有效任务
- 任务标题与描述
- 奖励预览
- 当前进度
- 状态：进行中 / 可领取 / 已领取 / 已过期
- 领取按钮

前台读取来源：

- Mongo：任务定义
- Redis：当前周期进度与可领取状态

玩家领取成功后需要同步刷新：

- 金币
- 强化石
- 天赋点
- 背包
- 任务状态

## 11. 与现有 Mongo 迁移能力的结合

本任务系统直接复用当前已完成的 Mongo 能力：

- `backend/internal/mongostore/`
- `admin_audit_logs`
- `domain_events`
- `system_logs`
- `archive.AsyncQueue`

建议新增：

- `backend/internal/mongostore/tasks.go`
- `backend/internal/mongostore/task_claim_logs.go`
- `backend/internal/mongostore/task_cycle_archives.go`

说明：

- 任务系统是新能力，不需要像 `boss_history`、`wall_messages` 那样做历史迁移
- 但它本身包含“周期归档到 Mongo”的内建迁移流程

## 12. 建议实施顺序

### 阶段 1：数据模型与仓储

- 定义任务配置结构
- 定义领取日志结构
- 定义周期归档结构
- 实现 Mongo 仓储与索引初始化

### 阶段 2：Redis 进度层

- 设计并实现 Redis 热键
- 实现周期 key 生成
- 实现 TTL 策略

### 阶段 3：任务服务

- 实现任务读取
- 实现事件推进
- 实现领取逻辑
- 实现并发防重

### 阶段 4：归档服务

- 实现过期周期扫描
- 实现周期归档
- 实现补归档

### 阶段 5：后台管理

- 新增后台接口
- 新增任务管理 Tab
- 支持周期结果查询

### 阶段 6：玩家前台

- 新增任务面板
- 展示任务、进度、状态
- 支持领取

## 13. 测试重点

必须覆盖：

- 日常跨天归档
- 周常跨周归档
- 限时任务到期归档
- 归档后 Mongo 能查到：
  - 已领取
  - 完成未领取
  - 未完成
  - 未参与
- 归档成功后 Redis 键可安全清理
- Redis TTL 不会早于归档
- 重复归档幂等
- 重复领取防重
- 并发领取只发奖一次
- 后台任务操作写入 `admin_audit_logs`
- 玩家领取写入 `domain_events`
- 异常归档写入 `system_logs`

## 14. 第一版默认假设

- 使用北京时间 `Asia/Shanghai`
- 周常按周一 00:00 刷新
- 第一版仅支持全服统一任务
- 第一版仅支持单条件单目标
- 第一版为手动领取
- 第一版装备奖励仅支持“指定模板 + 固定数量”
- 第一版不支持随机装备池
- 第一版不支持历史累计补算
- 第一版不支持任务到期后补领
- Redis 只保留当前和最近过期周期的热数据
- Mongo 是任务配置、领取历史、周期归档的最终事实来源
