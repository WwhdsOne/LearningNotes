# 实时状态链路与点击性能优化报告

## 1. 背景

这次优化的目标是提高投票墙在高并发场景下的吞吐能力，同时保持现有玩法和业务规则不变。  
优化前，系统的主要压力集中在两处：

- 实时更新依赖轮询和全量聚合，单次状态变化会重复触发大量读取与拼装。
- 后台和前台都存在按需不足、全量过多的问题，数据量上来后 CPU 和 Redis 压力会明显上升。

这次改动没有改变按钮玩法、Boss 规则、装备规则和后台管理边界，重点是把“怎么算状态、怎么推状态、怎么读 Redis”做了重构。

## 2. 本次优化解决了什么问题

### 2.1 优化前的主要问题

- 点击后需要重新聚合完整状态，重复计算较多。
- SSE 广播以完整状态为中心，公共信息和个人信息没有拆分。
- Redis 热路径里仍然存在按前缀扫描和大范围聚合。
- 后台玩家概览首屏读取过重，人数一多就会拖慢后台。
- Boss 点击链路虽然已经保证正确性，但仍有进一步压缩 Redis 往返和锁竞争的空间。

### 2.2 优化后的核心思路

本次优化把实时链路改成了“事件驱动 + 缓存快照 + 定向推送”的模式：

- 写操作成功后发布事件，而不是依赖高频轮询。
- 服务进程内缓存公共态和个人态，避免每次都重新聚合完整状态。
- SSE 按“公共态”和“个人态”拆开发送，减少无效广播和重复计算。
- Redis 增加显式索引，热路径优先走索引，不再频繁扫描键空间。
- 后台玩家数据改成分页和按需读取，避免首屏拉全量明细。

## 3. 主要改动说明

### 3.1 实时更新从轮询改成事件驱动

后端新增了 Redis 事件总线，业务写操作成功后会发布状态变更事件。  
服务启动后持续监听这些事件，再决定刷新哪些缓存、向哪些连接推送哪些数据。

这样做的好处是：

- 点击后不需要所有在线连接都重新跑一遍完整 `GetState`。
- 同一个变化只处理一次，再按需要广播。
- 实时链路更清晰，后续继续做风控、监控或扩展事件类型也更方便。

### 3.2 SSE 拆成公共态和个人态两类消息

现在 SSE 不再默认推送完整状态，而是拆成两种事件：

- `public_state`：所有用户共享的内容，例如按钮列表、排行榜、当前 Boss、公告等。
- `user_state`：只和当前昵称有关的内容，例如个人点击数、背包、装备栏、个人 Boss 伤害、最近掉落列表等。

这次拆分之后：

- 公共变化会广播给所有在线用户。
- 个人变化只会推送给对应昵称的连接。
- 前端收到消息后合并更新，不再依赖“每次都收到完整状态”。

这一步是本次性能优化的关键，因为它直接减少了重复计算和无意义网络传输。

### 3.3 Redis 热路径改成显式索引

为了避免频繁扫描 Redis 键空间，本次新增并使用了以下索引：

- `vote:buttons:index`
- `vote:equipment:index`
- `vote:players:index`

改动后的读取方式：

- 按钮列表优先通过按钮索引读取。
- 装备模板通过装备索引读取。
- 后台玩家列表通过玩家索引分页读取。

同时保留了一个低频兜底同步机制：如果有人直接手工往 Redis 写新按钮，服务会按 `button_poll_interval_ms` 配置的低频间隔做一次补索引同步，让这个按钮最终出现在页面上。

需要注意的是，这个配置现在表示“按钮索引低频兜底同步间隔”，已经不是全站状态轮询间隔。

### 3.4 后台玩家概览改成分页

`/api/admin/state` 现在只返回后台首页真正需要的摘要信息，不再默认带出所有玩家明细。  
玩家列表改成单独的分页接口，详情也改成按需读取。

这样做后，后台首屏更轻，玩家数量增加时更稳定，管理页不会因为全量聚合而明显变慢。

### 3.5 点击链路减少了不必要的回读和竞争

点击接口做了两类优化：

- 点击成功后不再额外回读一次完整状态，只返回这次点击直接相关的数据。
- Boss 点击结算继续保持原子性，但进一步压缩了执行路径，减少了额外争用。

这意味着单次点击的服务端工作量更小，响应链路也更短。

### 3.6 Lua 脚本改成 `SCRIPT LOAD + EVALSHA`

Boss 点击相关 Lua 脚本现在使用 Redis 的脚本缓存能力：

- 首次执行先 `SCRIPT LOAD`
- 后续直接 `EVALSHA`
- 如果 Redis 返回 `NOSCRIPT`，后端会自动重载并重试

同时，进程内还维护了一份脚本名到 SHA 的内存映射，避免每次都重新加载脚本内容。  
这能减少脚本传输开销，也能让热点 Lua 调用更稳定。

### 3.7 接入 `pprof`

后端已经接入 Go `pprof`，用于后续性能分析和压测定位。  
现在可以直接通过以下地址查看或采样：

- `/debug/pprof/`
- `/debug/pprof/profile`
- `/debug/pprof/heap`

这部分不会影响业务功能，但对后续继续压性能非常重要。

## 4. 对现有功能的影响

本次优化保持了以下业务行为不变：

- 按钮在平时仍然可以点击。
- Boss 活动期间，一次点击仍然同时记票并造成 Boss 伤害。
- 装备效果在平时点击和 Boss 战斗期间都继续生效。
- 当前版本仍然只支持单个当前世界 Boss。
- 后台仍然可以管理 Boss、装备、掉落池、按钮、公告和留言。

也就是说，这次优化主要改变的是系统内部的组织方式和传输方式，而不是用户看到的玩法规则。

## 5. 接口与配置上的变化

### 5.1 SSE 消息类型变化

SSE 从原来的“完整状态推送”改成了：

- `public_state`
- `user_state`

前端已经同步适配，不需要额外人工切换。

### 5.2 后台接口变化

新增了后台玩家分页相关接口：

- `GET /api/admin/players`
- `GET /api/admin/players/{nickname}`

### 5.3 配置语义变化

`button_poll_interval_ms` 的含义已经调整为：

“按钮索引低频兜底同步间隔”

它主要影响“直接手工写 Redis 新按钮后，页面多久能补同步出来”，不再表示公共态广播轮询间隔。

## 7. 这次优化带来的直接收益

从实现层面看，这次优化的直接收益主要有：

- 单次点击触发的状态聚合次数显著减少。
- 在线连接增加时，SSE 广播的额外成本更可控。
- Redis 热路径从“扫描和全量读”转向“索引和定向读”。
- 后台页面在玩家规模增长时更稳定。
- 后续做压测、热点排查和继续优化有了明确抓手。

简单说，这次改动把系统从“功能可用”往“能更稳地扛并发”推进了一步。

## 8. 当前边界与后续建议

这次优化已经完成了主链路重构，但仍有几个边界需要明确：

- 现在的点击风控仍然是单机内存限流，不适合多实例共享风控状态。
- `pprof` 虽然已经接入，但还需要结合真实压测结果继续分析热点。
- 如果后续在线人数和点击量继续上涨，下一步优先级最高的是点击防刷和访问控制，而不是继续扩大前端改造范围。

### 8.1 已补充的显示一致性修正

优化上线后，前台曾出现过“Boss 血量看起来回升一点”的显示问题。  
排查后确认，这不是 Redis 里真实血量回档，而是前端同时接收：

- 点击接口的立即返回
- SSE 的异步 `public_state`

在网络时序交错时，较旧的一条同 Boss 公共态可能晚于较新的点击结果到达，前端如果无条件覆盖，就会把血量短暂显示成更高值，看起来像“回档”。

当前已补充前端合并保护：

- 同一只、同一实例、且状态仍为 `active` 的 Boss
- 如果收到的后续消息 `currentHp` 更高
- 则视为旧消息，不再覆盖当前画面

因此，这次修正针对的是“显示一致性”，不是后端实际扣血逻辑变更。

建议的下一步工作顺序：

1. 基于当前版本做一次有规模的压测，拿到真实 `pprof` 数据。
2. 针对点击接口补更强的防刷策略，优先考虑不影响正常用户体验的方案。
3. 根据压测结果，再决定是否继续拆分更细的缓存和指标。

## 9. 结论

这次优化的价值不在于“新增了多少功能”，而在于把系统的实时状态链路、Redis 读取模式和后台聚合方式做了更适合并发场景的重构。  
在不改变现有玩法的前提下，系统现在更容易扩展、更容易定位性能问题，也更适合继续往线上高并发场景推进。
