## 2. 修改入口与字段含义

在 `backend/cmd/server/main.go` 的 `vote.NewStore(..., vote.StoreOptions{...})` 中修改：

1. `CriticalChancePercent`：基础暴击率（单位 `%`，取值会被裁剪到 `0~100`）。
2. `CriticalCount`：基础暴击加伤核心参数（整型）。

关键计算点：

1. 暴击率判定：
   - [backend/internal/vote/store.go](/Users/Learning/web/long/backend/internal/vote/store.go:1484)
   - [backend/internal/vote/store.go](/Users/Learning/web/long/backend/internal/vote/store.go:2459)
2. 普通点击链路下的暴击伤害：
   - [backend/internal/vote/store.go](/Users/Learning/web/long/backend/internal/vote/store.go:1545)
   - 公式：`CriticalDamage = NormalDamage + CriticalCount - 1`（至少不低于普通伤害）
3. Boss 多部位链路下的暴击伤害：
   - [backend/internal/vote/store.go](/Users/Learning/web/long/backend/internal/vote/store.go:1580)
   - 命中暴击后使用 `CritDamageMultiplier` 乘区，默认基线在 `baseCombatStats` 为 `1.0`

## 3. 推荐改法（按需求）

### 3.1 只改基础暴击率

只改 `CriticalChancePercent`，例如：

```go
store := vote.NewStore(redisClient, cfg.RedisPrefix, vote.StoreOptions{
	CriticalChancePercent: 12, // 基础暴击率 12%
	CriticalCount:         0,
}, nicknameValidator)
```

### 3.2 改“普通点击链路”的基础暴击加伤

只改 `CriticalCount`。例如希望“暴击比普通点击固定多 8 点”，设置：

```go
CriticalCount: 9, // 多伤 = CriticalCount - 1 = 8
```

说明：`CriticalCount <= 1` 会被归一到 `1`，等价于“无额外暴伤”。

### 3.3 改“Boss 链路”的基础暴击倍率

如果需求是“Boss 暴击时按倍率放大”，需要改 `baseCombatStats` 里的默认倍率：

```go
CritDamageMultiplier:  1.5, // Boss 暴击默认 1.5x
```

仅改 `CriticalCount` 不会直接提高 Boss 公式里的倍率乘区。

## 4. 一次性改动建议（避免认知偏差）

如果产品定义的“基础暴伤”要在“普通点击 + Boss”两条链路都体现，建议同时设置：

1. `CriticalCount`（影响普通点击链路的 `CriticalDamage`）
2. `CritDamageMultiplier`（影响 Boss 链路的倍率乘区）

否则会出现“同样写了基础暴伤，但两条链路体感不一致”的现象。

## 5. 修改后本地验证

建议最小验证命令：

```bash
rtk go test ./backend/internal/vote/...
```

如果仓库顶层已有统一检查入口，优先：

```bash
rtk make check
```

人工验证建议：

1. 无装备、无天赋新角色点击，观察 `criticalChancePercent` 与暴击触发频率是否符合预期。
2. 激活 Boss 后点击部位，确认暴击时伤害是否按预期倍率/加伤生效。
