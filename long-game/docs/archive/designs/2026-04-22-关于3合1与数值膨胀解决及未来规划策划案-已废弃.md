# 装备与英雄数值重构方案（删除 3 合 1，重做强化成长，并为特殊英雄与后续构筑扩展留口）

## Summary

当前数值膨胀的核心问题有 4 个：

- 2026-04-21 引入“装备取消升星上限”后，普通装备可以靠无限 3 合 1 持续抬高属性，直接击穿稀有掉落分层。
- 升星收益是随机加到点击 / 暴击 / 暴击率，导致成长不可控，玩家只要刷量就能把低阶装备养成高阶模板的替代品。
- 装备还叠加了原石强化，形成“升星一条线 + 强化一条线”的双重永久成长，数值上限缺失。
- 小小英雄觉醒同样是长期叠加，和装备一起把点击、暴击、Boss 伤害持续抬高，后期通胀失控。

新的总方向：

- 直接删除 3 合 1 系统，包括前端入口、接口、文档说明和历史 Redis 升星数据清理方案。
- 删除原有全部升星收益，不再保留任何由旧 3 合 1 驱动的成长效果。
- 装备强化与英雄觉醒继续保留，但改成“固定上限 + 明确公式 + 无大奖/无保底”的可控成长。
- 强化成本调整为：
    - 装备强化：20 -> 10
    - 英雄觉醒：25 -> 15
- 背包里继续保留装备名称上的 +N 显示，但其语义从“旧升星等级”改为“当前强化等级/阶段显示”。
- 背包里每个属性都要同时展示“当前值 + 强化增量”，例如 点击 12（+4），明确告诉玩家这件装备被改了多少。
- 小小英雄系统从这轮开始按“可扩展效果机制”设计，并额外为未来的战斗构筑预留底层支持能力，但本轮不启用流派玩法、不增加 UI 引导。

## 方案对比与选择

推荐采用方案 A：

- 方案 A：删除 3 合 1，强化/觉醒保留，但改为模板单独配置上限，并把英雄被动改成可扩展效果系统
    - 优点：既解决当前膨胀，又为后续特殊英雄和构筑玩法留出稳定扩展点。
    - 缺点：需要重写一部分英雄效果建模。
- 方案 B：删除 3 合 1，强化/觉醒保留上限，但英雄仍维持固定枚举字段
    - 优点：短期改动小。
    - 缺点：后续每出一个特殊英雄都要改核心战斗逻辑，扩展性差。
- 方案 C：删除 3 合 1，强化/觉醒递减收益，英雄特效继续硬编码
    - 优点：实现最快。
    - 缺点：同时留下数值和架构两类隐患。

本次采用：方案 A。

## Key Changes

### 1. 删除旧升星体系，但保留 +N 展示

- 删除 POST /api/equipment/{itemId}/synthesize 的产品定义。
- 前端移除“3 合 1 升星”按钮、说明文案、相关错误提示。
- 文档移除“无限升星”“升星不会生成新装备而是强化该账号下这个装备类型属性”等描述。
- 背包、穿戴区、掉落后展示中的装备名继续保留 木剑 +3 这类形式。
- +N 不再表示旧 3 合 1 升星次数，而是表示当前装备强化等级或强化阶段。
- 历史账号中旧 star_level / 旧升星收益不做迁移复用，直接彻底删除。
- Redis 清理目标是把历史 vote:user-equip-upgrade:<nickname>:<itemId> 里的旧升星字段和旧升星带来的数值全部清掉，不保留任何继承关系。
- 新的 +N 只从新强化体系重新累计，不接受旧账号自动换算。

### 2. 重做装备强化

- 装备强化消耗改为 10 原石。
- 去掉当前概率表、随机双加、大奖、保底进度。
- 每次强化时：
    - 读取该装备“基础属性 + 当前强化属性”的总和。
    - 取该总和的 1/4 作为本次强化增量基数。
    - 随机加到一个属性中。
    - 暴击率命中概率固定为 5%。
    - 剩余 95% 在点击 / 暴击之间分配。
- 每次成功强化时，装备的显示 +N 同步 +1。
- 每件装备有独立强化上限，由模板单独配置。
- 达到上限后不可继续强化，并给出统一前后端提示。
- 所有涉及“基于当前总属性值计算”的数值，统一采用向上取整规则：
    - 分解获得原石：取当前总属性值的 **1/6（向上取整）**
    - 强化与觉醒成长基数：取当前总属性值的 **1/4（向上取整）**
    - 前端做好每次强化的说明

### 3. 重做英雄觉醒

- 英雄觉醒消耗改为 15 原石。
- 去掉当前概率表、随机双加、大奖、保底。
- 每次觉醒时：
    - 读取该英雄“基础属性 + 当前强化属性”的总和。
    - 取该总和的 1/4 作为本次觉醒增量基数。
    - 随机加到一个属性中。
    - 暴击率命中概率固定为 5%。
    - 其余概率在点击 / 暴击 / 英雄被动值之间分配。
- 每个英雄模板单独配置觉醒上限。
- 达到上限后停止成长，保留英雄模板差异，不再靠无限觉醒拉爆后期。

### 4. 背包展示改为“当前值 + 增量值”双显

- 装备卡片继续显示装备名 +N。
- 每个属性都拆成两层语义：
    - 当前生效总值
    - 相对模板基础值的强化增量
- 展示格式固定为：
    - 点击 12（+4）
    - 暴击率 6%（+1%）
    - 暴击 9（+3）
- 如果某条属性没有被强化过，仍显示：
    - 点击 8（+0）
- 穿戴区和背包区保持同一展示规则。
- 掉落预览和 Boss 掉落池展示仍以模板基础属性为主，不混入玩家个人强化增量。

### 5. 小小英雄效果系统重构

- 不再把英雄能力长期绑定在少量硬编码字段上。
- 英雄模板层改为“基础属性 + effects[]”结构，每条 effect 至少包含：
    - type
    - value
    - params
    - displayName
    - description
- 当前已存在的普通能力继续以 effect 形式表达：
    - bonus_clicks
    - critical_chance_percent
    - critical_count_bonus
    - final_damage_percent
- 所有效果行为必须由 effect.type 驱动，禁止再出现 if heroType == xxx 这类角色硬编码分支。
- 未识别的 effect.type 不得导致主流程报错，应安全忽略并记录 warning。

### 6. 为未来战斗构筑预留基础设施（本轮不启用）

这部分要写入 docs，但本轮不实现完整玩法、不暴露给玩家：

- 当前项目仍是“单 Boss + 手动点击”的轻玩法结构。
- 本轮只补齐底层支持能力，确保未来可以平滑扩展不同战斗构筑，而不用再改核心战斗主流程。
- 这里的“构筑”只作为内部设计抽象，不进入当前接口命名、前端 UI、玩家文案。
- 未来可承载的方向包括但不限于：
    - 暴击向
    - 连击向
    - 斩杀向
- 当前玩家仍只感知为“不同英雄效果”，不会看到“流派”概念。

这一层基础设施需要满足 3 个条件：

1. effect 执行时可读取战斗上下文
   需要在设计文档中明确 BattleContext 概念，至少可提供：
    - currentHp
    - maxHp
    - comboCount
    - isCritical
    - damage
2. 支持多阶段触发
   需要把 effect 的触发阶段规范化，至少支持：
    - onClickBefore
    - onDamageCalculate
    - onCritical
    - onDamageFinal
3. 新 effect 可接入而不改主流程
   新增 effect.type 时，不应该去修改战斗主干逻辑，只需要新增 effect 处理器或注册映射即可。

文档中可用当前项目语境举例说明未来扩展方式，但明确标注“仅作为结构示例，本轮不启用”：

- 暴击向：
    - critical_damage_multiplier
    - 在 onCritical 阶段放大伤害
- 连击向：
    - combo_damage_bonus
    - 读取 comboCount 提供增伤
- 斩杀向：
    - boss_execute_by_max_hp
    - 在 Boss 低血量阶段触发特殊结算

同时明确本轮限制：

- 不做流派标签
- 不做流派 UI
- 不做流派数值平衡
- 不开放复杂多流派指数叠加组合
- 目标只是让系统结构允许未来做这些，而不是现在就做这些

### 7. 连击系统的扩展预留

- 本轮不强制把连击玩法做完整，但要在文档中预留数据与展示位。
- 为未来“动作游戏式连击数 + 增伤”保留最小设计边界：
    - 连击数属于战斗过程态，不写死在英雄模板基础属性里。
    - 英雄效果可以读取当前连击数参与伤害计算。
    - 前端 HUD 后续可增加明显的 Combo xN 展示和伤害浮字强化效果。

### 8. 文档与展示

后续 docs 需要新增一篇专门的平衡重构文档，建议内容结构：

- 背景：说明 2026-04-21 后无限升星带来的问题。
- 问题总结：列出普通装备刷穿超稀有、双成长线叠加、稀有度失效、后期成本失衡。
- 新规则：删除 3 合 1、新强化公式、固定上限、调整后成本。
- 显示规则：说明装备名 +N 继续保留，但改为强化等级展示；每条属性都显示总值和增量值。
- 英雄效果系统：说明从固定被动字段切到 effects[]。
- 构筑预留：用当前项目语言说明未来支持暴击向 / 连击向 / 斩杀向，但本轮不启用、不对玩家暴露。
- Redis 清理：给运维可直接执行的清理指令与风险提示。
- 前端变更：删除升星入口，新增强化说明按钮、升级动画、示例卡片。
- 平衡目标：普通装备不能再跨层替代高稀有装备；高阶掉落重新具备长期价值；后续特殊英雄不会再次把系统改乱。

## Public API / Types

需要在后续实现中明确这些接口/结构变化：

- 删除或废弃：POST /api/equipment/{itemId}/synthesize
- 保留并改规则：
    - POST /api/equipment/{itemId}/reforge
    - POST /api/heroes/{heroId}/awaken
- 装备模板新增：
    - 装备强化上限配置字段
- 装备状态新增或重定义：
    - 用于前端 +N 展示的强化等级字段
    - 每个属性对应的强化增量字段，或明确约定由前端用“总值 - 基础值”计算
- 英雄模板重构：
    - 从单一固定被动字段，升级为“基础属性 + effects[]”
- 战斗层预留：
    - BattleContext
    - effect hook 阶段枚举
    - effect handler 注册/分发机制
- 旧 starLevel 不做数据继承；如果字段继续保留，只能重定义为新强化显示等级，且历史值在迁移时清零或删除。

## Test Plan

需要覆盖这些场景：

- 删除 3 合 1 后，前端不再出现入口，后端接口不可用或返回统一废弃错误。
- 历史带 star_level 的账号在清理后不再获得任何旧升星收益，也不会继承旧 +N。
- 背包和穿戴区仍正确显示装备 +N，且该值只来自新强化。
- 背包和穿戴区每个属性都正确显示 总值（+增量）。
- 未强化属性正确显示 +0，不会缺字段或显示空白。
- 装备强化成功后，属性按新规则增长，同时 +N 正确加一。
- 英雄觉醒每次只加一个属性，且被动值进入随机池。
- 装备/英雄达到模板上限后，继续强化会被拒绝。
- 成本校验更新为装备 10、英雄 15。
- 英雄 effects[] 能正确解析并执行当前基础效果。
- 战斗上下文可被 effect 读取。
- effect 能在指定 hook 阶段执行。
- 未识别 effect.type 会被安全忽略，不会打断战斗主流程。
- 新增 effect handler 时，不需要改主战斗流程的分支结构。
- 老玩家清理旧升星数据后，最终战斗属性与文档规则一致，且不会残留旧膨胀值。
- 前端说明弹窗、示例展示、升级动画在成功/失败/到达上限三种状态都能正确表现。

## Assumptions

- 3 合 1 是彻底删除，不保留纯收集或占位玩法。
- 历史 Redis 升星数据直接删除，不迁移、不折算、不保留显示继承。
- 强化上限采用“模板单独配置”，不走全局统一值。
- 暴击率作为稀缺属性，强化/觉醒命中概率固定为 5%。
- 英雄觉醒成本按你最新要求固定为 15。
- 装备名称上的 +N 必须继续保留给玩家看，但只代表新强化等级。
- 背包属性展示必须显式告诉玩家“这件装备被强化改了多少”，默认用 总值（+增量） 形式展示。
- 构筑相关内容本轮只写入 docs 和结构设计，不进入当前 UI、接口文案和玩家认知。
- 特殊英雄本轮先做可扩展结构，不在本轮把斩杀、连击、残血增伤全部做完。
