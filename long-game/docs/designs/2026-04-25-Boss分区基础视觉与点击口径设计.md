# Boss 分区基础视觉与点击口径设计

## 背景

当前战斗页仍继承早期“按钮墙”模型：Boss 分区从已有按钮中抽取名称，点击分区时复用按钮点击链路。后续玩法已经转向 Boss 分区战斗，这套模型会带来几个问题：

- 部位展示依赖按钮名称，不利于给 Boss 部位配置独立名称和图片。
- 点击总数、Boss 伤害、官方挂机混在同一条按钮点击语义里。
- 前台视觉无法稳定围绕“软组织 / 重甲 / 弱点”三类部位做状态表现。

本阶段采用 A 方案：先落地 3 种部位类型的基础视觉和状态动画，天赋大特效仅预留结构，不在第一阶段完整实现。

## 第一阶段目标

- 删除前台战斗页对旧按钮池的展示依赖。
- Boss 分区直接展示后台配置的部位名称，支持可选小图。
- 部位类型仍由 `type` 表示：`soft`、`heavy`、`weak`。
- 手动点击次数改为每次手动点击只计 `+1`。
- Boss 伤害继续走现有装备、天赋、部位类型、护甲等公式。
- 官方挂机只结算 Boss 伤害，不增加点击次数。
- Boss 总血量继续由部位最大血量合计决定。

## 非目标

- 不在本阶段实现完整天赋大特效，例如暴风连击、结构崩塌、死兆收割、慢镜头、特殊音效。
- 不实现部位相邻联动机制。
- 不做旧按钮数据自动迁移为部位库。
- 不重写装备、天赋和 Boss 掉落规则。

## 数据模型

### BossPart 扩展

在现有 `BossPart` 上增加展示字段：

- `displayName`：部位显示名称，例如“左翼”“核心”“肩甲”。
- `imagePath`：可选小图路径或 URL，用于前台分区内展示。

保留现有战斗字段：

- `x`、`y`：5×5 坐标。
- `type`：部位类型，决定基础样式和伤害系数。
- `maxHp`、`currentHp`、`armor`、`alive`。

归一规则：

- 保存或激活 Boss 时，部位 `maxHp` 至少为 `1`。
- 新 Boss 实例激活时，所有部位重置为满血、`alive=true`。
- Boss 总血量为所有部位 `maxHp` 合计。
- Boss 当前血量为所有部位 `currentHp` 合计。

## 后台配置

Boss 模板编辑器保留 5×5 网格。

每个格子可配置：

- 部位名称。
- 部位类型：软组织、重甲、弱点。
- 部位血量。
- 护甲。
- 可选小图。

总血量输入保持只读，由部位血量合计展示。

如果部位名称为空，前台兜底显示类型文案，例如“软组织”“重甲”“弱点”。

## 前台战斗页

### 分区展示

战斗页不再从按钮列表抽取分区名称。每个 Boss 分区直接使用 `BossPart.displayName` 和 `BossPart.imagePath`。

分区内容优先级：

1. 小图存在时展示小图。
2. 展示部位名称。
3. 展示类型标签和血量。

### 三类基础样式

软组织：

- 标识色：`#4ADE80`。
- 2px 绿色实线边框。
- 背景：`rgba(74,222,128,0.15)`。
- 默认绿色微光呼吸，周期 2 秒。

重甲：

- 标识色：`#9CA3AF`。
- 3px 灰色金属质感边框。
- 背景：`rgba(156,163,175,0.2)`。
- 内阴影模拟厚度，默认静态，可带 4 秒金属扫光。

弱点：

- 标识色：`#EF4444`。
- 2px 红色脉动边框。
- 背景：`rgba(239,68,68,0.1)`。
- 默认红色脉动光晕，周期 0.8 秒。

### 基础状态

- 悬停：边框亮度提高，轻微放大。
- 按下：下沉 2px，内阴影加深。
- 受击：闪白后恢复。
- 低血量：低于 25% 时加深颜色，边缘出现紧张态。
- 被击碎：灰色滤镜、下陷、禁用交互。

本阶段先用 CSS 动画实现基础状态；粒子、全屏震动和音效留到后续天赋特效阶段。

### Debuff 图标区

Boss 总血条下方预留 Debuff 图标区。第一阶段可以渲染结构和空/亮状态，但不要求完整接入所有天赋来源。

预留图标：

- 护甲归零。
- 出血。
- 终结标记。
- 剥皮。
- 审判日已触发。

## 点击口径

### 手动点击

手动点击 Boss 部位时：

- 用户点击总数 `+1`。
- 排行榜按点击次数统计。
- Boss 伤害单独按现有战斗公式计算。
- 点击响应需要同时返回点击增量和 Boss 伤害结果。

这意味着“点击次数”和“伤害数字”不再是同一个值。

### 官方挂机

官方挂机继续由服务端托管。

挂机点击时：

- 不增加用户点击总数。
- 不进入点击排行榜。
- 仍按 Boss 战斗公式结算伤害。
- 仍发布 Boss 状态变化，让前台看到部位血量和总血量变化。

### 历史兼容

旧按钮接口和按钮管理需要在实现计划中拆解处理。本阶段目标是前台战斗和 Boss 分区不再依赖按钮；是否彻底删除后台按钮页、Redis 按钮索引和旧 API，需要在实现计划里按测试覆盖分步清理。

## 实现分层建议

1. 后端数据模型：扩展 `BossPart` 展示字段，补归一和序列化测试。
2. 后端点击链路：拆分手动点击统计和 Boss 伤害结算，挂机不计点击次数。
3. 后台编辑器：增加部位名称和小图配置。
4. 前台战斗页：移除按钮抽名逻辑，改用部位字段渲染。
5. 前台样式：落三类基础样式、低血量、受击、击碎状态。
6. 回归测试：覆盖手动点击、挂机、部位展示、总血量合计、基础样式标识。

## 风险点

- 当前点击链路以 `button slug` 为入口，删除按钮依赖会影响手动点击票据、WebSocket 点击、HTTP 点击兜底和官方挂机目标。
- 点击次数和伤害数值拆分后，`ClickResult.Delta` 的语义需要重新命名或明确分字段，避免前端把伤害当点击增量展示。
- 官方挂机不计点击次数后，排行榜和 Boss 伤害榜的差异会更明显，前端文案需要避免误解。
- 旧按钮管理页和按钮 API 是否彻底删除，可能影响已有测试和历史文档，需要单独清理。

## 验收标准

- 后台可给每个 Boss 部位配置名称、类型、血量、护甲和可选小图。
- 战斗页 5×5 分区不再展示旧按钮名称和票数。
- 手动点击任一存活部位后，用户点击总数只增加 1。
- 手动点击造成的 Boss 伤害仍按现有战斗公式扣减部位血量和总血量。
- 官方挂机造成 Boss 伤害，但用户点击总数不增加。
- 软组织、重甲、弱点三种基础样式和低血量、击碎状态可见。
- 现有前端测试、后端测试和构建通过。
