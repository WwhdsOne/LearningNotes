# 2026-04-22 玩家账号密码与 JWT 接入

## 改造目标

- 玩家侧从“只靠昵称即身份”切到“昵称 + 密码 + JWT 会话”。
- 首次使用某个昵称时，允许直接为该昵称设置密码并认领账号。
- 后端写接口在启用玩家认证后，优先使用服务端登录态，不再信任客户端传入的 `nickname`。
- 后台增加“重置某玩家密码”的运营入口。

## 后端变化

- 新增 `playerauth` 服务层：
  - 玩家密码通过 `bcrypt` 哈希后存入 Redis。
  - 登录成功后签发 JWT。
  - JWT 通过 `HttpOnly` Cookie 维持会话。
  - 密码重置后会让旧 JWT 失效。
- 新增玩家认证接口：
  - `POST /api/player/auth/login`
  - `POST /api/player/auth/logout`
  - `GET /api/player/auth/session`
- 新增后台密码重置接口：
  - `POST /api/admin/players/{nickname}/password/reset`
- 前台写接口在开启玩家认证后统一要求玩家先登录：
  - 点击
  - 装备操作
  - 英雄操作
  - 外观购买/装备
  - 公共留言
- 前台读接口的个人态也优先从玩家会话解析昵称：
  - `GET /api/buttons`
  - `GET /api/shop`
  - `/api/events`

## 数据与配置变化

- Redis 新增玩家认证键：
  - `vote:player-auth:<nickname>`
- 账号记录至少包含：
  - `nickname`
  - `password_hash`
  - `created_at`
  - `updated_at`
  - `password_reset_at`
- 现有游戏数据仍继续沿用原有昵称主键，不做整库迁移：
  - `vote:user:<nickname>`
  - `vote:user-inventory:<nickname>`
  - `vote:user-hero-inventory:<nickname>`
  - 其他同类玩家资产键
- Consul 配置新增：
  - `player_auth.jwt_secret`
  - `player_auth.jwt_ttl_seconds`

## 前端变化

- 公共页登录区从“输入昵称上墙”改为“输入昵称和密码登录/首次认领”。
- 前端不再把昵称持久化到 `localStorage` 作为身份来源。
- 页面初始化时先请求 `/api/player/auth/session` 判断是否已有登录态。
- 登录成功后再拉取个人态和建立带身份的 SSE 连接。
- 退出登录后会清空本地个人态视图，并调用后端登出接口。
- 管理后台玩家概览列表新增“重置密码”按钮，走后台管理接口完成密码重置。

## 兼容策略

- 为避免把现有测试和无认证场景全部打断，HTTP 层保留了一层兼容逻辑：
  - 未注入 `PlayerAuthenticator` 的场景，仍可按旧的 `nickname` 参数工作
  - 注入后，生产路径统一以玩家 JWT 会话为准
- 现有昵称敏感词校验继续保留在后端执行，首次认领时同样生效。
