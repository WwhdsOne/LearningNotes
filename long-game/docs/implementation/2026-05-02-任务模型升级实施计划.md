# 任务模型升级实施计划

> **执行方式**：按测试驱动逐步实施；每一阶段先补失败测试，再写最小实现。

**目标：** 将任务系统从 `taskType + conditionKind` 升级为 `eventKind + windowKind`，兼容现有旧数据，并提供一个可直接填写 MongoDB URI 与集合名的独立迁移脚本。

**架构：** 后端先引入新字段并提供新旧双向兼容映射，运行时统一按 `eventKind` 匹配事件、按 `windowKind` 计算周期；前端后台编辑器切换为新字段组合；迁移脚本独立连接 Mongo，批量为旧任务定义与任务归档补齐新字段。

**技术栈：** Go、Hertz、MongoDB、Vue 3、Vitest、miniredis

---

## 文件职责

- `backend/internal/vote/task_types.go`
  - 新增任务事件类型、累计窗口类型及兼容字段。
- `backend/internal/vote/tasks.go`
  - 统一事件匹配、周期键生成、活动判定与 TTL 逻辑。
- `backend/internal/vote/task_admin.go`
  - 统一保存校验、归档周期计算、归档写入字段。
- `backend/internal/mongostore/tasks.go`
  - 读写任务定义时支持新旧字段共存与回退映射。
- `backend/internal/mongostore/task_cycle_archives.go`
  - 读写任务归档时持久化 `event_kind`、`window_kind` 并兼容旧字段。
- `backend/internal/httpapi/router_test.go`
  - 覆盖后台任务接口对新字段的透传与兼容。
- `backend/internal/vote/tasks_test.go`
  - 覆盖新模型下的进度、校验、归档核心行为。
- `frontend/src/pages/admin/state.js`
  - 前端任务表单默认值、归一化与展示标签升级。
- `frontend/src/pages/admin/useAdminPageActions.js`
  - 新模型校验、保存载荷与兼容处理。
- `frontend/src/components/admin/AdminTaskTab.vue`
  - 任务编辑器和列表筛选切换到“行为类型 + 累计窗口”。
- `frontend/src/pages/TaskPage.vue`
  - 玩家端任务标签展示切换到新文案。
- `frontend/src/pages/AdminPage.taskTab.test.js`
  - 覆盖后台任务页接线是否包含新控件和新状态。
- `backend/cmd/migratetaskmodel/main.go`
  - 独立迁移命令，参数直填 MongoDB URI、数据库名、任务集合名、归档集合名。
- `backend/cmd/migratetaskmodel/main_test.go`
  - 覆盖参数解析、旧字段映射与文档补写逻辑。

## 执行步骤

### 阶段一：后端模型与兼容层

1. 在 `backend/internal/vote/tasks_test.go` 增加失败测试：
   - 旧任务 `daily_clicks` 能映射为 `click + daily`
   - 新任务 `click + fixed_range` 能累计进度
   - `fixed_range` 非法时间窗会被拒绝
2. 运行：
   - `go -C backend test ./internal/vote -run 'Test(ListTasksForPlayer|RecordTaskEvent|SaveTaskDefinition)'`
3. 修改 `task_types.go`、`tasks.go`、`task_admin.go`：
   - 新增 `TaskEventKind`、`TaskWindowKind`
   - 统一兼容归一化函数
   - 统一 `currentTaskCycleKey` / `archiveCycleKeyForTask`
4. 再次运行同一组测试，确保转绿。

### 阶段二：Mongo 存储兼容

1. 在 `backend/internal/httpapi/router_test.go` 或新增 `mongostore` 测试中补失败测试：
   - 新字段可以透传保存
   - 只有旧字段的文档也能读成新模型
2. 修改 `backend/internal/mongostore/tasks.go` 与 `task_cycle_archives.go`：
   - 写入新旧字段
   - 读取时优先新字段，缺失时回退旧字段映射
3. 运行：
   - `go -C backend test ./internal/mongostore ./internal/httpapi`

### 阶段三：前端后台与玩家端

1. 在 `frontend/src/pages/AdminPage.taskTab.test.js` 先补失败断言：
   - 后台任务编辑器存在 `eventKind`
   - 存在 `windowKind`
   - `fixed_range` 条件下展示时间输入
2. 修改 `state.js`、`useAdminPageActions.js`、`AdminTaskTab.vue`、`TaskPage.vue`：
   - 表单改为“行为类型 + 累计窗口”
   - 列表与归档文案改为新模型
   - 保持旧接口返回时也能展示
3. 运行：
   - `npm --prefix frontend run test`

### 阶段四：独立迁移脚本

1. 新增 `backend/cmd/migratetaskmodel/main_test.go` 失败测试：
   - 参数必须显式传 `--mongo-uri`、`--db`、`--tasks-collection`
   - 归档集合名可自定义
   - 旧文档映射后补写 `event_kind`、`window_kind`
2. 新增 `backend/cmd/migratetaskmodel/main.go`：
   - 不依赖 `config`
   - 直接通过命令行参数连接 Mongo
   - 支持 `plan`、`migrate`、`verify`
3. 运行：
   - `go -C backend test ./cmd/migratetaskmodel`

### 阶段五：总验证

1. 运行后端相关测试：
   - `go -C backend test ./internal/vote ./internal/mongostore ./internal/httpapi ./cmd/migratetaskmodel`
2. 运行前端测试：
   - `npm --prefix frontend run test`
3. 如有必要，补充一次迁移命令示例到最终说明中。
